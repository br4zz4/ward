# Ward CLI design and initial implementation

> TLDR: Design and implement `ward`, an opensource Go CLI for hierarchical secrets management using SOPS+age, replacing company-branded wehive terraform wrapper.

**Status:** proposed
**Created:** 2026-04-20
**Owner:** @oporpino

---

## Context

WeHive's secrets management is coupled to a company-branded `wehive` CLI that:
- Cannot be maintained openly or adopted by others
- Has no secrets hierarchy — flat per environment only
- Duplicates values between SOPS files and GitHub Actions secrets
- Uses `.sops.yaml` config with no project identity

`ward` is the opensource replacement — generic enough for any team, named after the inner mechanism of a lock (controls access without being explicit about "secrets").

## Objectives

- Hierarchical secrets via SOPS+age with a tree structure defined by YAML content
- Project-owned config format (`ward.yaml`) and file format (`*.ward`)
- `ward get`, `ward list`, `ward exec`, `ward init`, `ward edit` commands
- Conflict detection with clear error messages pointing to source files and line numbers
- `ward list` shows merged tree with file origin and line number per value
- `ward exec` with optional anchor file — loads only that file's ancestry line
- Configurable merge strategy per project (`override` | `merge` | `error`)
- Replace `wehive terraform` in commons and CI pipelines

## Design

### ward.yaml (committed, no secrets)

```yaml
encryption:
  engine: sops+age
  key_env: WARD_AGE_KEY       # env var holding age private key (CI)
  key_file: .ward.key         # local key file (gitignored)

merge: override               # override | merge | error

sources:
  - path: ../commons/secrets  # relative path to external secrets
  - path: /org/infra/secrets  # absolute path
```

### *.ward files (SOPS+age encrypted YAML internally)

Structure is free — content defines the hierarchy, not the filename or path.
Filenames and folder structure are convention only.

```yaml
# company.ward
company:
  name: "company name"
  sectors:
    one:
      name: "sector 1"

# company/sectors/one/staging.ward  (more specific — overrides ancestor)
company:
  sectors:
    one:
      name: "override sector 1"
      staging:
        database_url: "postgres://staging..."
        redis_url: "redis://..."
```

### Hierarchy and merge

**Content is the source of truth.** Filenames and folder structure are convention only — the dot-path comes from the YAML keys, not the file path.

When a file is passed as anchor, ward loads only that file's line of ancestry (parents, not siblings):

```
company.ward                          ← loaded (ancestor)
company/sectors.ward                  ← loaded (ancestor)
company/sectors/one.ward              ← loaded (ancestor)
company/sectors/one/staging.ward      ← loaded (anchor)

company/sectors/one/production.ward   ← NOT loaded (sibling)
company/sectors/two/staging.ward      ← NOT loaded (cousin)
```

Without anchor, ward loads all sources from `ward.yaml` sources and fails if ambiguous keys exist at the same level.

Ancestry is determined by matching common dot-path prefixes in the YAML content — not by filesystem path.

### Conflict detection

Conflict = same dot-path defined in two files at the same ancestry level.
Error lists **all** conflicting sources with file and line:

```
ward: conflict detected on key "company.sectors.one.database_url"
  → company/sectors/one.ward:4
  → company/sectors/one/staging.ward:2

  ward cannot resolve this automatically.
  resolve by:
    1. removing from one of the files
    2. moving to a common ancestor if shared across environments
```

`--filter` / `--env` equivalent: pass the anchor file to scope which branch is loaded, avoiding sibling conflicts.

### Commands

**`ward get <dot.path>`**
Returns the merged subtree at that path (decrypted, YAML output).

```bash
ward get company.sectors.one
# returns merged subtree for that node
```

**`ward list [dot.path]`**
Shows the merged tree with source file and line for each value (decrypted for display only).

```
$ ward list company.sectors.one

company.sectors.one
  name: "override sector 1"          ← company/sectors/one.ward:3
  staging
    database_url: "postgres://..."   ← company/sectors/one/staging.ward:2
    redis_url: "redis://..."         ← company/sectors/one/staging.ward:3
  production
    database_url: "postgres://..."   ← company/sectors/one/production.ward:2
```

Without dot.path — shows the full merged tree from all sources.

**`ward exec [anchor.ward] -- <cmd>`**
Merges secrets (anchor optional), injects leaf nodes as env vars (uppercased, `__` as separator), runs cmd.

```bash
ward exec company/sectors/one/staging.ward -- terraform apply
# injects: COMPANY__SECTORS__ONE__STAGING__DATABASE_URL=...
#          COMPANY__SECTORS__ONE__STAGING__REDIS_URL=...
```

Without anchor — loads all sources, fails if ambiguous keys exist at the same level.

**`ward init`**
Generates `ward.yaml` template and first `*.ward` file for a project.

**`ward edit <file.ward>`**
Opens decrypted file in $EDITOR, re-encrypts on save.

## Changes

### New repo: `ward` (this repo)

```
cmd/ward/           ← CLI entrypoint (cobra)
internal/config/    ← parse ward.yaml
internal/secrets/   ← discover *.ward files, load + merge
internal/sops/      ← thin wrapper over SOPS+age
```

### Commons changes (wehive.tech/.commons)

- Replace `lib/wehive/terraform.sh` with thin shell helper calling `ward` binary
- Update `shell/tasks/terraform/secrets-init.sh` — generate `ward.yaml` + `*.ward` files
- Update `shell/tasks/ci/terraform/apply.sh` — use `ward exec` instead of `SOPS_AGE_KEY_FILE`
- Update `make/shared/terraform.mk` — replace `wehive terraform` with `ward exec -- terraform`
- Replace `shell/tasks/terraform/templates/.sops.yaml.template` with `ward.yaml.template`
- Remove `lib/wehive/terraform.sh`

### CI changes (wehive.tech/pipelines)

- Remove GitHub Actions secrets duplicating SOPS values
- Only secret needed: `WARD_AGE_KEY` (age private key)
- Add `ward` binary install step to CI runner

## How to verify

1. `ward exec company/sectors/one/staging.ward -- env` shows correct env vars injected
2. `ward exec company/sectors/one/production.ward -- env` shows production vars, no staging leakage
3. `ward list company.sectors.one` shows merged tree with file origins
4. Conflict in same ancestry line → non-zero exit with sources listed
5. `ward exec` without anchor with ambiguous keys → non-zero exit
6. CI applies terraform using only `WARD_AGE_KEY` secret
